原文： Claude Code 分层多Agent架构篇 🏗️ 一、架构概述 1.1 整体架构设计 Claude Code采用 分层多Agent架构 ，通过主Agent与SubAgent协作处理复杂任务。核心流程如下： - 用户请求 → 主Agent(nO函数)判断是否调用Task工具 → 是→Task工具(p_2对象)创建SubAgent(I2A函数) → 进行Agent生命周期管理、并发执行协调(UH1函数) → 结果合成器(KN5函数)返回结果 - 否→直接处理工具调用并返回结果 1.2 核心技术特点 完全隔离的执行环境 ：每个SubAgent在独立上下文中运行 智能并发调度 ：支持多Agent并发执行，动态负载均衡 安全权限控制 ：细粒度的工具权限管理和资源限制 高效结果合成 ：智能的多Agent结果聚合和冲突解决 弹性错误处理 ：多层错误隔离和恢复机制 🔧 二、Task工具Agent实例化机制 2.1 Task工具核心定义 Task工具是多Agent架构的入口点，其核心实现包含： - 常量定义 ：cX = &ldquo;Task&rdquo;（工具名称） - 输入Schema ：CN5，包含description（3-5字任务描述）和prompt（Agent执行任务） - 完整对象结构 ：p_2，包含动态描述生成、名称、描述、输入Schema、核心执行函数等 // Task工具完整对象结构核心部分 p_2 = { async prompt({ tools: A }) { return await u_2(A); }, // 动态描述生成 name: cX, // "Task" async description() { return "Launch a new task"; }, inputSchema: CN5, async * call({ prompt: A }, context, J, F) { /* Agent启动和管理逻辑 */ }, isReadOnly() { return true; }, isConcurrencySafe() { return true; }, // 其他属性... } 2.2 动态描述生成机制 Task工具描述由u_2函数动态生成，包含可用工具列表及使用规则： - 排除自身 ：过滤Task工具防止递归调用 - 使用场景 ：推荐在搜索关键词/文件且不确定匹配结果时使用Agent工具 - 禁用场景 ：读取特定文件路径、搜索特定类定义等情况应使用Read/Glob/Grep等工具 - 使用注意 ：支持并发启动多Agent、结果需用户显式总结、Agent调用无状态需详细任务描述 2.3 SubAgent创建流程 SubAgent由I2A函数负责实例化，流程包括： 1. 生成唯一Agent ID ：调用VN5()生成 2. 创建初始消息 ：包含任务提示 3. 获取配置信息 ：模型配置(qW())、资源配置(RE())、默认模型(J7()) 4. 生成系统提示 ：基于selectedModel和工作目录 5. 执行Agent主循环 ：调用nO函数（主Agent循环） 6. 处理结果 ：提取文本内容、计算token使用量、保存对话历史 🔒 三、SubAgent执行上下文分析 3.1 上下文隔离机制 每个SubAgent通过SubAgentContext类实现完全隔离： - 独立工具集合 ：过滤Task工具防止递归调用 - 继承权限上下文 ：获取父上下文的工具权限 - 资源限制 ：最大执行时间(5分钟)、工具调用次数(50次)、token限制(100000) - 独立中断控制器 ：AbortController实例 - 独立工具使用状态 ：setInProgressToolUseIDs集合 3.2 工具权限继承和限制 SubAgent继承主Agent基本权限但受额外限制，由ToolPermissionFilter类管理： 工具类型 限制措施 Write 最大文件大小5MB，需验证，禁止写入系统目录 Edit 每次调用最多10处修改，需备份，禁止危险代码模式 Bash 超时120秒，禁止rm -rf/sudo等命令，仅允许ls/cat等安全命令 WebFetch 限制域名(docs.anthropic.com等)，响应大小5MB，超时30秒 3.3 资源分配的独立性 SubAgentResourceMonitor类监控资源使用： - 时间监控 ：检查执行时间是否超过5分钟限制 - Token计数 ：累计token使用量，超过100000则抛出错误 - 工具调用计数 ：限制最多50次工具调用 - 文件操作和网络请求 ：分别记录操作次数 🔀 四、并发执行协调机制 4.1 并发执行策略 Task工具支持两种执行模式，由parallelTasksCount配置决定： - 单Agent模式 ：直接调用executeSingleAgent执行 - 多Agent并发模式 ：调用executeParallelAgents，创建多个相同任务的Agent，最大并发数10 并发执行流程： 1. 创建多个Agent任务 → 并发执行(UH1函数) → 收集结果 2. 调用KN5函数生成合成提示 → 启动合成Agent → 生成最终结果 4.2 并发调度器实现 UH1函数实现异步生成器的并发执行，核心逻辑： - 包装生成器为Promise，追踪执行状态 - 控制最大并发数，初始启动maxConcurrency个任务 - 使用Promise.race等待任一生成器结果，完成后启动新任务 - 持续处理直到所有生成器完成 4.3 Agent间通信和同步 通过结构化消息系统实现，主要消息类型： - PROGRESS ：进度更新，包含toolUseID和消息数据 - RESULT ：执行结果，包含Agent索引、内容、工具使用次数等 - ERROR ：错误信息 - STATUS_UPDATE ：状态更新 🔄 五、Agent生命周期管理 5.1 Agent创建和初始化 AgentInstanceManager管理生命周期，包含状态枚举： - INITIALIZING(初始化) → RUNNING(运行中) → WAITING(等待中) → COMPLETED(完成)/FAILED(失败)/ABORTED(中止) 创建流程：生成Agent ID → 创建隔离上下文 → 初始化资源监控器 → 记录消息历史 5.2 资源管理和清理机制 AgentResourceCleaner负责资源清理： - 清理任务 ：执行注册的清理函数 - 临时文件 ：删除Agent创建的临时文件 - 网络连接 ：关闭Agent建立的网络连接 5.3 超时控制和错误恢复 超时控制 ：AgentTimeoutController默认5分钟超时，超时后调用abort() 错误恢复 ：AgentErrorRecovery实现最多3次重试，带指数退避策略(1s, 2s, 4s) 📋 六、工具白名单和权限控制 6.1 SubAgent工具白名单 SubAgent仅可访问预定义工具集合： 工具类别 具体工具 文件操作 Read, Write, Edit, MultiEdit, LS 搜索工具 Glob, Grep 系统交互 Bash(受限) Notebook工具 NotebookRead, NotebookEdit 网络工具 WebFetch(受限), WebSearch 任务管理 TodoRead, TodoWrite 计划模式 exit_plan_mode 被禁用工具：Task(防止递归)及其他敏感工具 6.2 工具权限验证器 ToolPermissionValidator类实现严格权限验证： 1. 检查工具是否在白名单 2. 执行工具特定权限检查（如Bash命令过滤、WebFetch域名验证） 3. 应用全局安全策略 4. 记录工具使用日志 6.3 递归调用防护机制 RecursionGuard类防止递归调用： - 禁止SubAgent调用Task工具 - 限制调用深度（默认maxDepth=3） - 限制每层Agent数量（默认maxAgentsPerLevel=5） 📊 七、结果合成和报告机制 7.1 多Agent结果收集逻辑 MultiAgentResultCollector类管理结果收集： - 按Agent索引存储结果 - 统计总token数、工具调用次数、执行时间、错误数 - 提供获取所有结果、成功结果的方法 7.2 结果格式化和合并 KN5函数负责结果合并，流程： 1. 按Agent索引排序结果 2. 提取各Agent文本内容，格式化为&rdquo;== AGENT X RESPONSE ==&rdquo; 3. 生成合成提示，要求综合所有Agent见解、解决矛盾、呈现统一解决方案 7.3 智能总结生成机制 IntelligentSynthesizer类根据任务类型生成总结： - 代码分析 ：合并代码块、提取技术细节、生成建议 - 问题解决 ：排序解决方案、提取实施步骤、识别潜在问题 - 研究任务 ：综合研究发现、整理来源、生成结论 7.4 结果一致性保证 ResultConsistencyValidator类验证结果一致性： - 事实一致性检查：检测矛盾陈述 - 代码一致性检查：验证命名规范、接口定义一致性 - 推荐一致性检查：确保建议不冲突 - 计算置信度：基于一致性和Agreement Bonus 🔁 八、主Agent循环机制深度分析 8.1 nO函数核心实现 nO函数是Agent系统核心，实现完整对话循环： 1. 检查上下文压缩需求，调用wU2函数压缩 2. 调用语言模型(wu函数)获取响应 3. 处理模型fallback（如Opus 4限制时切换模型） 4. 提取工具调用并协调执行(hW5函数) 5. 处理工具结果，递归调用nO继续循环 8.2 工具执行协调器hW5 hW5函数协调工具执行： - 调用mW5函数按并发安全性分组工具 - 并发执行安全工具(uW5函数) - 串行执行非安全工具(dW5函数) 8.3 单个工具执行MH1 MH1函数执行单个工具调用： 1. 标记工具使用状态 2. 检查工具是否存在 3. 检查中断信号 4. 执行工具调用(pW5函数) 5. 处理错误并清理状态 🧩 九、实际混淆代码实现还原 9.1 核心函数映射表 关键混淆函数映射（部分）： 混淆函数名 实际功能 nO executeMainAgentLoop（主Agent循环） I2A launchSubAgent（SubAgent启动器） u_2 generateTaskDescription（Task描述生成） KN5 synthesizeMultipleAgentResults（结果合成） UH1 concurrentExecutor（并发执行调度器） hW5 coordinateToolExecution（工具执行协调器） MH1 executeToolWithValidation（工具执行引擎） 9.2 还原的核心数据结构 包括AgentExecutionContext（执行上下文）、ToolExecutionResult（工具结果）、AgentResultData（Agent结果）等接口定义，规范了数据交互格式。 9.3 完整的Task工具对象还原 基于代码分析还原了TaskToolObject完整结构，包含prompt生成、名称、描述、输入Schema、核心执行函数等详细实现。 📝 补充细节 关键常量值 ：MAX_CONCURRENT_TOOLS=10（最大并发工具数），MAX_PARALLEL_AGENTS=3（最大并行Agent数），MAX_CONTEXT_LENGTH=40000（最大上下文长度） 安全限制 ：Bash工具禁止20+危险命令，WebFetch限制4个安全域名，Write工具限制5MB文件大小 性能优化 ：实现上下文压缩机制，当消息长度超过阈值时自动压缩历史对话 错误处理 ：多层级错误隔离，单个Agent失败不影响整体系统，支持自动重试和模型降级
