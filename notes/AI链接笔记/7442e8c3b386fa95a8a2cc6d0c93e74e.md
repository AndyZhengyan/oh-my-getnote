原文： I Reverse Engineered ChatGPT's Memory System, and Here's What I Found! 🧠 ChatGPT记忆系统核心发现（背景） 研究背景与方法 - 核心问题 ：ChatGPT如何存储和检索用户信息，实现无缝对话体验？ - 研究方法 ：通过对话逆向工程（非官方文档），发现其记忆系统由 四个层级 构成，而非传统向量数据库或RAG架构。 - 关键结论 ：该系统以轻量级、分层注入的方式平衡个性化与性能，优于传统复杂检索系统。 🏗️ ChatGPT上下文结构全景（基础框架） 完整提示词结构 ChatGPT每次接收消息时的上下文包含7个模块，其中后5个与记忆相关： 模块编号 模块名称 核心功能 [0] System Instructions 高层行为定义（非记忆重点） [1] Developer Instructions 安全规则（非记忆重点） [2] Session Metadata 会话环境适配（临时） [3] User Memory 长期事实存储（核心档案） [4] Recent Conversations Summary 近期对话摘要（兴趣连续性） [5] Current Session Messages 当前会话滑动窗口（完整上下文） [6] Your latest message 用户最新输入 🔍 四层记忆架构深度解析（核心内容） (一) Session Metadata（会话元数据） 临时环境适配层 - 定义 ：会话开始时注入的环境信息， 会话结束后不保留 。 - 关键组成 ： - 设备类型（桌面/移动）、浏览器及用户代理 - 大致位置/时区、订阅级别（如ChatGPT Go） - 使用模式（近30天活跃18天）、模型使用分布（如gpt-5占49%） - 设备环境（深色模式、屏幕尺寸900×1440等） - 作用 ：使模型响应适配当前使用场景。 (二) User Memory（用户记忆） 显式长期事实层 - 定义 ：存储用户稳定信息的 持久化档案 ，注入所有未来对话。 - 存储触发条件 ： 1. 用户明确指令（如“记住这个”） 2. 模型检测符合OpenAI标准的事实（如姓名、职业）+用户隐含同意 - 典型内容示例 ： - User's name is Manthan Gupta. - Previously worked at Merkle Science and Qoohoo (YC W23). - Prefers learning through videos, papers, and hands-on work. - Built TigerDB, CricLang, Load Balancer, FitMe. 管理方式 ：支持直接增删（如“从记忆中删除…”）。 (三) Recent Conversations Summary（近期对话摘要） 轻量级兴趣连续性层 - 定义 ：用户近期对话的 摘要列表 ，仅包含用户消息片段。 - 格式与特征 ： 1. <Timestamp>: <Chat Title> |||| user message snippet |||| 仅总结用户消息，不含助手回复 约15条摘要，反映近期兴趣而非详细内容 优势对比（vs传统RAG） ： 维度 ChatGPT摘要法 传统RAG系统 实现方式 预计算轻量摘要直接注入 嵌入所有消息+相似度搜索+全量上下文 性能 低延迟、低token成本 高延迟、高计算资源消耗 功能 提供兴趣连续性 提供详细历史上下文 (四) Current Session Messages（当前会话消息） 实时滑动窗口层 - 定义 ：当前会话的 完整消息历史 ，未摘要化。 - 特性 ： - 基于token数量的滑动窗口（未明确token上限） - 旧消息在达到上限时被移除，但记忆事实和摘要层不受影响 - 作用 ：保证当前会话内的逻辑连贯性。 ⚙️ 记忆系统协同机制（工作流程） 用户发送消息后的处理流程 ： 1. 环境适配 ：注入Session Metadata（设备、订阅等） 2. 长期档案 ：加载User Memory（姓名、职业目标等33项事实） 3. 兴趣延续 ：附加Recent Conversations Summary（近15个对话主题） 4. 上下文整合 ：拼接Current Session Messages（当前会话完整历史） 5. 响应生成 ：结合用户最新消息输出结果 💡 关键洞察与补充细节 反常识发现 ：未使用RAG或向量数据库，而是通过 人工标注式记忆+轻量摘要 实现个性化，大幅降低复杂度。 记忆触发逻辑 ：模型会主动检测并存储符合标准的事实（如职业目标），用户无需显式指令。 数据量参考 ：User Memory约存储33项核心事实，Recent Conversations Summary包含约15条用户消息摘要。 局限性 ：牺牲历史对话细节换取效率，无法检索具体过往对话内容。
