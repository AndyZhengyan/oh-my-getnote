原文： 升级 Claude Code 后钱包在流血？我扒了 11 个版本的源码找到了原因 📰 事件背景与核心争议 事件起源 - 触发点 ：Moonshot AI工程师@Hx1u0于2026年2月13日发文称，Claude Code的&rdquo;神秘更新&rdquo;导致Kimi K2.5的Cache命中率降低，后经修复恢复正常。 - 核心指控 ：@sun_hanchi转发猜测Anthropic在Claude Code中植入代码，当检测到非Claude模型时，通过篡改system prompt空格破坏缓存机制，构成 蓄意反竞争行为 。 为何重要 - Prompt Cache机制 ：LLM API通过缓存相同前缀的system prompt+对话历史（prefill）降低重复计算成本。Cache命中时仅需为新增内容付费，未命中则需重复支付system prompt的token费用。 - 经济影响 ：Claude Code的system prompt长达 数千到上万个token ，若每日100轮对话缓存失效，用户将为同一段prompt重复付费100次。 🔍 逆向工程调查 代码获取与分析方法 - 通过 npm pack @anthropic-ai/claude-code 获取v2.1.41版本代码，核心分析7667行、11MB的混淆JavaScript文件 cli.js 。 关键搜索结果 搜索目标 关键词 结果 模型检测逻辑 if model != "claude" 、条件分支 ❌ 未找到 随机注入字符 Math.random 、 nonce 、 salt +prompt ❌ 未找到 空格篡改 whitespace 、 pad 、 space +system prompt ❌ 未找到 时间戳注入 Date.now 、 timestamp +system prompt ❌ 未找到 第三方特殊处理 openai 、 kimi 、 deepseek +条件判断 ❌ 未找到 结论 ：客户端代码中 不存在基于模型名称篡改system prompt的逻辑 ，阴谋论不成立。 🔧 缓存失效的技术原因 (一) System Prompt动态内容 动态拼接机制 ：由身份定义、系统规则等十几个函数动态生成，包含 版本信息 （如 BUILD_TIME: "2026-02-13T01:11:22Z" ）。 高频发版影响 ：Claude Code 几乎每天一版 （两周14个版本），每次发版 BUILD_TIME 变化导致system prompt改变，缓存失效。 (二) A/B测试控制 通过GrowthBook实验系统控制部分prompt内容，如： tengu_bergotte_lantern ：控制语气简洁/详细 tengu_vinteuil_phrase ：切换完全不同的prompt路径 结果 ：同一用户不同时间可能收到不同prompt，破坏缓存一致性。 (三) Claude模型专属缓存特权 私有缓存机制 ：Claude模型通过 scope: "global" 实现 跨session、跨用户全局缓存共享 ， ttl: "1h" 实现1小时有效期。 第三方模型劣势 ：仅依赖基础prefix逐字节匹配缓存，system prompt任何微小变化（如空格）即导致缓存失效。 (四) 缓存失效监控系统 代码中存在完整的cache break监控，记录失效原因并上报遥测数据，反证非故意破坏。 📅 版本考古与关键转折点 核心版本时间线 版本范围 关键变化 对第三方缓存影响 v2.1.0~v2.1.22（2026-01-28前） system prompt结构简单稳定，无A/B测试和分块缓存 ✅ 正常工作 v2.1.23（2026-01-29） 引入分块缓存、全局缓存flag、prompt分块逻辑 ⚠️ 开始受影响 v2.1.30+（2026-02-03+） 引入system prompt A/B测试变体 ❌ 命中率下降 v2.1.38+（2026-02-10+） 新增adaptive-thinking等特性，prompt复杂度增加 ❌ Moonshot公开回应 结论 ：缓存问题源于Anthropic对Claude模型的持续优化（全局缓存、分块缓存等），其副作用破坏了第三方模型依赖的prefix matching机制，属于 &ldquo;自家优化的附带伤害&rdquo; 。 💡 用户自救指南 1. 锁定版本 推荐版本 ：锁定v2.1.22（2026-01-28发布），为分块缓存引入前的最后一个&rdquo;安全&rdquo;版本。 安装命令 ： npm install -g @anthropic-ai/claude-code@2.1.22 代价 ：失去v2.1.23+新功能，但缓存效率显著提升。 2. 经济影响测算 基于2月3日~10日真实数据（总token 375.7M，缓存命中占91.1%）： 场景 Input侧等效成本（按10%缓存折扣计算） 成本变化 缓存正常 66M等效token 基准 缓存全部失效 374M等效token 暴涨至5.7倍（多花470%） 命中率从91%→50% ~180M等效token 成本翻倍 经济损失示例 ：若第三方模型input单价¥1/百万token，月均多支出约 ¥1,200+ 。 3. 监控API账单 关注指标： - cache_creation_input_tokens ：新建缓存消耗token（全价） - cache_read_input_tokens ：缓存读取token（折扣价） - 警示信号 ：前者远大于后者表明缓存失效严重。 4. 选择适配的Provider 优先选择已修复该问题的Provider（如Moonshot），其通过归一化处理忽略无关动态字段，提升缓存命中率。 5. 关注开源替代方案 开源工具链可审计代码，避免&rdquo;黑盒行为&rdquo;，适配第三方模型更友好。 🌐 AI Infra中立性思考 核心命题 ：当通过A公司客户端调用B公司模型时，A公司的技术决策直接影响用户体验与成本。 - 现状 ：Claude Code为Claude模型深度优化（全局缓存、分块缓存等），但未考虑第三方模型适配，导致体验劣势。 - 隐忧 ：类似&rdquo;Embrace, Extend, Extinguish&rdquo;策略，通过非主动适配使第三方模型体验逊于原生模型。 - 呼吁 ： 开源、可审计的AI工具链 是保障透明度的关键，可避免&rdquo;黑箱&rdquo;争议。 📝 补充细节 &ldquo;五篇大文章&rdquo;未涉及 ：本文聚焦技术分析，未提及产业政策相关内容。 分析局限性 ：仅限于客户端代码，不包含Anthropic服务端行为。 数据来源 ：基于Claude Code v2.1.0~v2.1.41共11个版本的静态代码逆向分析。
